# CLAUDE.md — Global Mineral Deposit Globe

## What we're building

An interactive 3D globe that plots the world's known mineral deposits as glowing,
commodity-colored points on a dark planet, filterable by commodity. A reader can spin
the Earth, toggle commodities on and off, and click any deposit to see its details.

This is a **static single-page web app** — no backend, no database, no server. The
deposit data is processed once into a JSON file that ships with the app. The whole thing
can be hosted on any static host (GitHub Pages, Netlify, Vercel).

This is the first of two related builds. Keep the data pipeline and component structure
clean enough that a later project can reuse them to draw **trade-flow arcs** between
deposits and processing facilities (a "supply-chain globe"). Don't build that now — just
don't paint us into a corner.

## Success criteria

In priority order:

1. **It looks intentional and beautiful, not templated.** This is the whole point of the
   project. The globe is a hero object in deep space; the deposit points are a constellation
   wrapping the planet. Follow the `globe-viz` skill (in `.claude/skills/globe-viz/`) for the
   visual system — it is not optional styling advice, it is the brief.
2. It runs smoothly — the globe spins at 60fps and filtering feels instant.
3. The data is honestly labeled (correct source attribution and data vintage — see Data).
4. It works on a laptop screen and degrades gracefully on mobile.

## Tech stack

- **Vite + React + TypeScript** — fast dev server, clean build.
- **react-globe.gl** (wraps globe.gl, which wraps three.js) — the 3D globe and points layer.
  Chosen over deck.gl for v1 because it produces a polished globe with almost no boilerplate.
- **Plain CSS with custom properties** (CSS variables) for the design tokens — no UI kit.
  The token values live in `.claude/skills/globe-viz/references/design-system.md`; mirror
  them into `src/styles/tokens.css`.
- No state library needed — React `useState`/`useMemo` is enough.

Install: `npm create vite@latest . -- --template react-ts`, then
`npm i react-globe.gl three`.

Run: `npm run dev`. Build: `npm run build`.

## Data

### Sources (download these; do not hardcode fake data)

- **MRDS — USGS Mineral Resources Data System**: <https://mrdata.usgs.gov/mrds/>
  The best *global* collection of deposit locations (name, lat/lng, commodity, deposit type).
  Download the full dataset (CSV or shapefile) from that page.
- **Critical Minerals in Ores (CMiO) / USGS critical-mineral deposit layers**, also at
  <https://mrdata.usgs.gov/> — use these to enrich or focus the critical-mineral subset.

### Important honesty note (affects credibility — do not skip)

USGS **stopped systematically updating MRDS in 2011**, and it is the best *global* set but
is historical, with known duplicate and quality issues. The app **must** state this in the
attribution footer, e.g.:
`Data: USGS MRDS (global, not updated since 2011) + Critical Minerals in Ores. Locations are historical and approximate.`
Never present this as a live or complete picture. Getting this right matters more than it seems.

### Pipeline (`scripts/prepare-data.ts` or a Python script — your call)

1. Load the raw download.
2. Keep only rows with valid numeric latitude/longitude.
3. Normalize the commodity field (MRDS `commod1` / commodity columns are messy) into our
   fixed commodity taxonomy (see the palette in the skill: gold, copper, lithium, cobalt,
   nickel, rare earths, zinc, uranium, silver, other). Map synonyms and codes to these buckets.
4. **For v1, filter to critical minerals + the major base/precious metals** to keep the point
   count in the low thousands. The full MRDS is ~300k points and will choke react-globe.gl's
   points layer — see Performance. State the filter applied in the UI.
5. Write `public/deposits.json` as a compact array:
   `[{ "name": string, "lat": number, "lng": number, "commodity": string, "depositType"?: string }, ...]`
   Round coordinates to 4 decimals to shrink the file.

## Architecture

```
src/
  main.tsx
  App.tsx                 // layout shell: globe + panel + footer
  components/
    DepositGlobe.tsx      // react-globe.gl wrapper, points config, click handling
    CommodityPanel.tsx    // left control panel: toggles + live counts
    Legend.tsx            // commodity color key
    DepositCard.tsx       // detail card shown on point click
  data/
    commodities.ts        // the commodity taxonomy + colors (from the skill palette)
    useDeposits.ts        // loads public/deposits.json, applies active filters
  styles/
    tokens.css            // design tokens mirrored from the skill
    globals.css
public/
  deposits.json           // generated by the pipeline
```

Single source of truth for commodity → color → label lives in `src/data/commodities.ts`.
Both the points layer and the legend read from it. Never hardcode a commodity color twice.

## Build phases (do them in order; get each looking right before moving on)

**Phase 1 — the globe exists and is beautiful.** Dark-space background, styled Earth,
atmosphere glow, all deposits rendered as commodity-colored points, gentle auto-rotate on
load. No interactivity yet. Stop and make this genuinely look good before anything else.

**Phase 2 — filtering.** Left panel with one toggle per commodity, each showing a live count
and its color swatch. Toggling updates the globe instantly. Add the legend and the
attribution footer.

**Phase 3 — detail + polish.** Click a point → detail card (name, commodity, type,
coordinates). Hover state on points. Reduced-motion handling, keyboard focus, mobile layout.
Empty/loading states written in the interface's voice.

**Later (NOT now): graduation to deck.gl.** If we ever want to render the *full* ~300k-point
MRDS, or add trade-flow arcs for the supply-chain globe, switch the rendering layer to
deck.gl `GlobeView` + `ScatterplotLayer`/`ArcLayer`, which handle large data on the GPU.
Keep `DepositGlobe.tsx` isolated so only it changes.

## Performance

- react-globe.gl's HTML/points layers are fine for a few thousand points, sluggish past
  ~10–20k. Stay in the low thousands for v1 via the data filter.
- If points still feel heavy, set `pointResolution` low and avoid per-frame React re-renders
  (memoize the points array; don't recompute it on every state change).
- Don't ship a multi-megabyte JSON. Filter and round coordinates in the pipeline.

## Constraints

- No secrets, no API keys — all data is public and baked in at build time.
- Keep dependencies minimal: Vite, React, react-globe.gl, three. Add nothing else without a
  reason.
- The design tokens are the law. If a color or font isn't in the skill's design system,
  don't introduce it.
